Funcionalidades clave

📝 Bloques arrastrables

Texto, encabezados, listas, tablas, imagenes, archivos, embeds. Cada bloque se puede arrastrar para reordenar. Slash commands (/) para insertar bloques rapidamente.

📂 Jerarquia de paginas

Paginas dentro de paginas, estilo Notion. Arbol de navegacion en sidebar izquierdo. Drag & drop para reorganizar la estructura.

🗃️ Bases de datos

Tablas con propiedades tipadas (texto, numero, fecha, select, multi-select, persona). Vistas: tabla, kanban, galeria, lista.

👥 Colaboracion

Multiples editores simultaneos. Indicador de quien esta viendo/editando. Historial de cambios para revertir.

🔗 Links cruzados

Mencionar @pagina, @tarea, @persona. Las menciones se actualizan si la pagina cambia de nombre. Backlinks automaticos.

📤 Plantillas

Plantillas predefinidas: SOP, Meeting Notes, Product Brief, Weekly Report. El usuario puede crear plantillas propias.

Enfoque tecnico

Editor: TipTap (basado en ProseMirror) — el mismo motor que usan Notion, GitBook, y Linear. Soporta bloques custom, drag & drop, collaborative editing con Y.js.

Almacenamiento: Firestore — cada documento es un doc con campo `content` (JSON del editor), `title`, `parent_id`, `created_by`, `updated_at`, `permissions`.

Real-time: Y.js + WebSocket para edicion colaborativa. Firestore listeners para cambios de estructura (nuevo doc, renombrar, mover).

Referencia open source: AFFiNE (bloque editor), Outline (wiki con jerarquia). No copiar — entender patrones UX.

Funcionalidades clave

💬 Canales con permisos

Canales publicos y privados. Admin controla quien puede ver/escribir. Canales por tienda, area, o tema. Threads para mantener orden.

📩 Mensajes directos

DMs 1:1 y grupales. Indicador de presencia (online/offline). Estado personalizado. Read receipts opcionales.

🎤 Notas de voz

Grabar audio inline en el chat. Reproductor con waveform. Duracion maxima 5 min. Almacenamiento en Firebase Storage.

📎 Archivos y previews

Drag & drop archivos. Preview de imagenes, PDFs, links. Previews enriquecidos de links internos (docs, tareas, dashboards).

🔔 Notificaciones Web

Web Push Notifications API. El usuario recibe notificacion aunque tenga la pestana cerrada. Configuracion por canal (todo, solo menciones, silenciar).

🔗 Integraciones internas

Mencionar @vega para que la IA responda en el canal. Compartir links de Docs, Tareas, Dashboard con previews automaticos.

Enfoque tecnico

Real-time: WebSocket nativo (ws) o Socket.io. Cada canal es un "room". Reconexion automatica.

Almacenamiento: Firestore para persistencia de mensajes. Coleccion `channels/{channelId}/messages`. Index por `timestamp`.

Notificaciones: Web Push Notifications API + Service Worker. Firebase Cloud Messaging (FCM) como proveedor.

Audio: MediaRecorder API del browser. Upload a Firebase Storage. Almacenar URL en el mensaje.

Server: Next.js API routes para REST (crear canal, listar mensajes). WebSocket server separado en Railway o como custom server de Next.js.

Referencia: Mattermost (channels + threads UX), Rocket.Chat (open source Slack).

Funcionalidades clave

📅 Vistas multiples

Dia, Semana, Mes. Vista de agenda (lista). Drag & drop para mover eventos entre dias/horas.

🎥 Google Meet

Crear evento con link de Google Meet automatico. Integracion via Google Calendar API. Un click para unirse.

🔗 Tareas vinculadas

Los deadlines de Tareas aparecen automaticamente. Crear tarea desde el calendario. Colores por tipo (reunion, deadline, evento).

👥 Calendario de equipo

Ver disponibilidad del equipo. Filtrar por persona. Calendarios superpuestos para encontrar huecos.

🔔 Recordatorios

Notificacion web 15 min antes. Configurable por evento. Integrado con el sistema de notificaciones del Chat.

🔄 Sync bidireccional

Eventos de Google Calendar se muestran en Grand Line y viceversa. El usuario no tiene que duplicar info.

Enfoque tecnico

API: Google Calendar API v3. OAuth2 ya configurado en Grand Line (Firebase Auth con Google). Scopes: `calendar.events`, `calendar.readonly`.

Almacenamiento: Firestore para eventos internos (no de Google). Merge con eventos de Google en el frontend.

UI: FullCalendar.js o construccion custom con CSS Grid. La vista de mes es un grid 7 columnas, semana es 7 columnas con filas de horas.

Google Meet: Al crear evento con tipo "videollamada", se usa `conferenceDataVersion: 1` en la API para generar link automatico.

Como funciona

⌨️ Activacion

⌘K (Mac) / Ctrl+K (Windows) — abre el modal de busqueda global en cualquier parte de la app.

Tambien hay un icono de busqueda en el sidebar que abre el mismo modal. Siempre accesible, sin importar en que modulo estes.

🔍 Alcance de busqueda

Busca en paralelo en todas las fuentes:
• Docs — titulo y contenido
• Chat — mensajes y archivos
• Tareas — titulo, descripcion, comentarios
• Modulos — navegar a cualquier seccion
• Personas — buscar por nombre de equipo

Enfoque tecnico

Indexado: Firestore full-text search tiene limitaciones. Dos opciones:
  1. Algolia — search-as-a-service, rapido, faceted search. Sync con Cloud Functions.
  2. Typesense — open source, self-hosted en Railway. Sin costo de licencia.

Recomendacion: Empezar con busqueda basica (Firestore queries + client-side filter). Migrar a Typesense cuando el volumen crezca.

UX: Resultados agrupados por tipo (como en el mockup). Navegacion con flechas del teclado. Enter para ir al resultado. Busqueda debounced (300ms).

Esquema de Firestore (nuevas colecciones)

Coleccion Documento Campos clave Modulo
channels/{channelId} Canal de chat name, type (public/private/dm), members[], team_id, created_at Chat
channels/{id}/messages/{msgId} Mensaje text, author_uid, timestamp, attachments[], thread_id, reactions{} Chat
docs/{docId} Documento title, content (TipTap JSON), parent_id, team_id, created_by, icon, cover Docs
docs/{id}/versions/{vId} Version content, edited_by, timestamp Docs
events/{eventId} Evento title, start, end, all_day, attendees[], google_event_id, meet_link, team_id Calendar
search_index/{id} Indice type (doc/msg/task), title, body_preview, source_id, team_id, updated_at Search
notifications/{uid}/items/{nId} Notificacion type, title, body, source_url, read, timestamp Global

Multi-tenant (sin cambios)

Todas las nuevas colecciones usan team_id como campo de scope. El patron effectiveUid = profile?.team_id || user?.uid se mantiene identico.

Security Rules de Firestore validan que el usuario pertenezca al team antes de leer/escribir. Channels privados validan membership adicional.

WebSocket para Chat en tiempo real

Opcion A: Server dedicado en Railway

Node.js + Socket.io corriendo en Railway (~$5/mes). Ventajas: sin limites de Vercel (serverless no soporta WS persistentes). Desventaja: otro servicio que mantener.

Recomendado

Opcion B: Firestore real-time listeners

Sin WebSocket server. Cada cliente escucha onSnapshot en la coleccion de mensajes. Ventajas: cero infra adicional. Desventaja: mas reads de Firestore, mayor latencia (~500ms vs ~50ms).

Recomendacion: Enfoque hibrido

Fase 1 (MVP): Firestore real-time listeners. Sin WebSocket. Funciona perfecto para equipos pequenos (<20 personas). Zero infra adicional. Listo en 1 semana.

Fase 2 (Scale): Migrar a WebSocket cuando el volumen de mensajes justifique. El frontend no cambia mucho — solo el hook de conexion.
